GNet Integration API

Introduction

GNet Platform
GNet platform is an Open Distribution Network for ground transportation providers to send and receive reservations electronically. GNet acts as a messenger service that transmits reservations into target reservation systems and keeps these reservations synchronized until trip completion.

GNet is a B2B platform interfacing mainly with software companies who service Ground Transportation and Corporate Travel Service providers.

Every Transportation company is associated with a unique GRiDD ID. This ID is used to identify the providing company and proper routing to the target back-office reservation system.

GNet API services are RESTful.
Actors

  • Requester- An entity sending a reservation request to be fulfilled
  • Provider- An operator who will fulfill the reservation.
  • GNet - Platform to exchange reservations and provide real-time tracking.
  • Driver - Chauffeur
  • Vendor - Software Vendors - Fasttrak Cloud, Livery Coach, Transdev, Hudson etc.
  • Passenger - Primary passenger or traveler
  • Account - Client account or requester account number

    • Requester Pre-requisites A requester must be connected to the GRiDD Platform™ and be assigned a GRiDD ID for reservations to be exchanged. Requester might be a Transportation Provider who is sending a reservation via their reservation system to their affiliate provider who utilizes a different reservation system.
    Requester may also be a Travel Management software and can use any choice of technology to integrate with the GRiDD API.
    Provider Pre-requisites Provider must be connected to the GNet Platform™ and be assigned a GRiDD ID for reservations to be exchanged. Provider may use reservation systems that the GRiDD platform is integrated into.

    Sending Reservations

    One standard format needed for trip payloads to be sent over GNet to any target dispatch system. Follow the recommended guidelines and you will have a glitch free integration.

    1. affiliateReservation section

    This is the header section of the payload. It contains the GNet member Id of the sender, the receiver and their reservation numbers. affiliateReservation.action field serves as the command field as to what the payload is for. The valid values for the action data field are NEW, UPDATE, and CANCEL.

    affiliateReservation.action , affiliateReservation.requesterId, affiliateReservation.providerId, and affiliateReservation.requesterResNo are required data items when submitting a payload to GNet.

If submitting an UPDATE or CANCEL, then affiliateReservation.providerResNo is also required.

    2. Addresses

    Regular Addresses

    Regular addresses are passed in two different formats in every address section.

  • locationType="ADDRESS"
  • Full one line address with street address, city , state, zip included. (no apartment numbers, no notes should be present)
  • Same address broken down to Street address, City, State, Zip, and Country.

    • Airport Addresses

    Airport addresses require IATA codes for airports and airlines. Use of IATA codes is required, otherwise airport trips cannot be understood.

  • locationType="AIRPORT"
  • 3 Character IATA code for for the airport to be used in address field.
  • 2 Character IATA code for the airline to be used in the airlineCode field.
  • Actual Flight number to be in the flightNumber field, do not use TBD, etc in that field, leave empty of unknown.
  • If the airport is a FBO, then set FBO field to "True" in that leg. Airline code 00 is also another indication that the pickup or drop off is a FBO.

    • 3. Passengers

    There must be at least one passenger in the passenger list. The passenger array is only limited to the target system's limits and can handle as many passengers in the list as needed. Passenger First Name, Last Name are required. Phone numbers are recommended, but if none is supplied, and target system requires it, then GNet will use the Sender's main office number as an alternate number.

    4. Notes

    There different note sections in each payload.

  • Each address section has a specialInstructions field which is the notes section for that address.
  • Each trip payload has a specialInstructions at the root level which holds the main trip notes.
  • Notes array at the root level holds Driver Notes and Affiliate Notes

    • 5. Pickup Time format

    Pickup, Drop off, and Stops Date/time are local to pickup location and contain no timezone setting. The format for these fields are YYYY-MM-DDTHH:mm:ss , ex: 2023-08-08T19:00:00

    Receiving Reservations

    If an existing API does not exist for GNet integration, then you can consume GNet's object model for receiving trips. GNet requires 1 web service on the receive side to accept NEW, UPDATE or CANCEL requests all in one web service. In order to provide status updates for incoming trips, you would need to call providerUpdateByResNo webservice on GNet. This endpoint should be called when there is change in Status, Driver Info, Vehicle Info, Charge line items, and Total

    GNet team will work with your technical team to determine how to communicate with your webservice, capture confirmation numbers, as well as error messages in case a trip has to be rejected.

    Web service receiving trip payloads from GNet must look at affiliateReservation section to determine if the payload is requesting a NEW trip, or its an UPDATE or a CANCEL Request for an existing trip.

    affiliateReservation.action tells the web service you are building, what needs to be done. NEW means its a new trip, UPDATE is an update of an existing trip, and CANCEL is a request to CANCEL a trip

    affiliateReservation.requesterId is the GNet ID of the sending company affiliateReservation.requesterResNo is the reservation# of the sending company affiliateReservation.providerId is the GNet ID of the receiving company affiliateReservation.providerResNo is the reservation# of the receiving company, this field will be populated if action=UPDATE or CANCEL

    Webhook callbacks

    GNet platform provides real-time callbacks on any changes that occur on sent trips. These changes can be Status Codes, Chauffeur info, Vehicle Info, Closing Costs and Total Amount during trip close process. This is a POST call made from GNet to sending party's webservice listener. This is not configurable from your side, you will need to let the GRiDD team know when your webservice is ready, and our team will configure it in for you.

    Send Reservation Request

    This service is called by the Requester. This service is called for All NEW reservations, and Updates. The difference between NEW and UPDATE reservations are :

    1. affililateReservation.action can be set to : 'NEW' or 'UPDATE'
    2. TransactionId must be supplied with every payload
    3. if submitMode="wait" the API call will be synchronous and reponse time will be longer. transactionId is always provided. when submitMode="wait", a reservationId is also provided.

    Response will be a unique Transcation ID within the GRiDD Platform for reference. Action="NEW" & status=“NEW” & Status : BOOKING_REQUEST This is an Asynchronous service call. Response can be a valid Transaction ID or an error caught by the rules engine, other responses will be provided via a callback at a later time.

    sendTrip

    URI Parameters
    versionThe value for version should be V1
    200 OK
